# Auth UX Audit

Generated by auth UX audit agent. FE-only — see wiki-be's companion audit for backend findings.

## Findings by flow step

### Discovery

No findings. The topbar auth button (`#auth-btn-home` / `#auth-btn-index` / `#auth-btn-content`)
is present and visible on every page context checked, labeled "Login" when logged out, opens the
modal to the login panel on click, and correctly focuses `#auth-login-email` on open.

### Register form

#### [CONFIRMED][MAJOR] Register submit fires duplicate POST on rapid double-click

- **Flow step:** Register form
- **File:** `js/auth.js:346-352` (submit handler), `Auth.register` (`js/auth.js:225-240`)
- **Repro:** Fill valid email + password meeting all checklist rules → rapid double-click
  `#auth-reg-submit` → 2 × `POST /api/v1/auth/register` fired (confirmed via Playwright network
  log, mocked backend).
- **Impact:** User can trigger duplicate registration attempts; a slower real backend increases
  the window. Second request likely surfaces a confusing "already registered" style error tied to
  the user's own first click. The same gap was independently confirmed on login, forgot-password,
  and reset-password submit handlers (2× requests each on rapid double-click) — see their
  respective sections; this is a systemic pattern across all four forms, not register-specific.
- **Fix direction:** Disable submit button synchronously in each submit handler before awaiting
  the async call (`register`, `login`, `forgotPassword`, `resetPassword`), re-enable on error. A
  shared helper (e.g. wrap the handler body) would fix all four call sites at once.

#### [MINOR] No loading/pending state shown during any auth request

- **Flow step:** Register form / Login / Forgot / Reset password
- **File:** `js/auth.js` (all flow methods: `login`, `register`, `forgotPassword`,
  `resetPassword`, `resend`) — button text never changes to "Logging in…" etc, no spinner element
  in any `index.html` auth panel.
- **Repro:** Verified via code read (Phase 1) — no live network throttling tool was available, but
  confirmed no code path anywhere in `auth.js` mutates button text/adds a spinner class between
  calling `api.auth.*` and its resolution.
- **Impact:** On a slow network, user has no feedback that anything happened after clicking
  submit — combined with the double-submit gap above, users are likely to click again, compounding
  that issue.
- **Fix direction:** Add a pending/disabled state with a text change ("Logging in…") tied to the
  in-flight promise, alongside the double-submit fix above (same code path).

#### [MINOR] No client-side timeout for hung network requests

- **Flow step:** Register form / Login / Forgot / Reset password
- **File:** `js/api.js:20-31` `_request` — fetch has no `AbortController`/timeout; only a
  try/catch around the fetch call itself for outright network errors (DNS failure, connection
  refused).
- **Repro:** Verified via code read only — no `AbortController`, no `setTimeout`-based abort
  anywhere in `_request`.
- **Impact:** A hung request (slow/degraded backend) leaves the user staring at an unresponsive
  form indefinitely with no escape hatch besides closing the modal. Compounds the no-loading-state
  finding above.
- **Fix direction:** Add a client-side timeout (e.g. 15-20s) via `AbortController` that surfaces a
  "Request timed out, try again" error state.

### Post-register state

No findings beyond what's covered in Register form / Resend verification. Confirmed live: after a

sent a verification link…"), the resend button is visible, and closing the modal leaves the

false-positive logged-in state).

### Email verification

#### [CRITICAL] First-time visit to an email verify/reset link silently loses the token via service-worker install reload

- **Flow step:** Email verification (also affects Forgot/reset password link flow)
- **File:** `js/auth.js:301-318` `handleBootParams()` strips `mode`/`token` from the URL via
  `history.replaceState` immediately on read; `js/app.js:505-539` service worker registration — on
  a browser with no prior SW for this origin, the very first `controllerchange` event fires
  unconditionally `location.reload()` with no guard against a fresh/first install.
- **Repro:** Fresh browser context (no prior visit/SW), navigate directly to
  `/?mode=verify&token=xyz` (simulating a clicked email link). Confirmed via Playwright: at t=0 the
  URL is `.../?mode=verify&token=validtoken123` and `#auth-modal` briefly has class `auth-modal`
  (open); by ~300ms the DOM node reads as briefly null then reappears as `auth-modal hidden`,
  `navigator.serviceWorker.controller` becomes truthy, and `location.href` is now bare
  `http://localhost:8790/` — the verify panel never renders its result, the modal ends up closed,
  and the user lands on a default logged-out home page with zero indication their verification (or
  reset) link did anything. Confirmed the underlying verify/reset logic itself is correct by
  re-running with service workers blocked — with SW disabled, the same URL correctly opens the
  verify-result panel with "Email verified! You can log in now." and a working back-to-login
  button.
- **Impact:** This hits every genuinely first-time user clicking their verification email link (or
  reset-password link) on a device/browser that has never loaded the site before — precisely the
  normal real-world case for email verification. The token is silently discarded; the user has no
  idea whether verification succeeded, failed, or did nothing, and the modal is simply closed with
  no error message. They must return to their email and click the link again — if the token
  happened to be single-use, it may now be reported as "already used" the second time even though
  it was never actually consumed by the FE on the first, silently-swallowed attempt.
- **Fix direction:** Guard the `controllerchange` → `location.reload()` with a check that skips the
  reload when it's the very first activation (`!navigator.serviceWorker.controller` at
  registration time, i.e. no previous controller to replace), or defer/skip the reload entirely
  when `location.search` contains `mode`/`token` params that haven't been consumed yet. Simplest
  fix: only reload on `controllerchange` if there was already a controller before this
  registration (i.e. this is an update, not a first install).

Confirmed live and working correctly (with SW blocked, isolating this from the bug above):
expired-token messaging ("This verification link has expired."), already-used-token messaging
("This account is already verified."), and the missing-token case (`?mode=verify` with no `token`
param correctly does nothing — no modal, no request, per the `if (!mode || !token) return;` guard
in `handleBootParams`).

### Resend verification

#### [MAJOR] Resend-verification gives zero user feedback and has no debounce/rate-limit guard client-side

- **Flow step:** Resend verification
- **File:** `js/auth.js:242-248` `Auth.resend()` — swallows all responses/errors silently, no
  toast, no button state change; `_wireModalInputs` click handler (`js/auth.js:353-357`) has no
  in-flight guard or disable.
- **Repro:** Reach verify-pending panel → click "Resend email" → button text unchanged, no toast,
  no disabled state. Click twice more rapidly → 3 total `POST /auth/resend-verification` fired
  with no client-side throttling.
- **Impact:** User has absolutely no confirmation the resend worked — button looks identical
  before and after every click. Combined with no rate-limit UX, a user unsure if it worked will
  likely click repeatedly, hammering the endpoint. If the backend does rate-limit, that failure is
  silently swallowed too (comment in code says "generic 200 either way; ignore") so the user never
  learns why a second email never arrives.
- **Fix direction:** Show a toast/inline confirmation on click ("Verification email sent"),
  disable the button for a cooldown window (e.g. 30-60s) after each click, and surface a friendly
  message if a real rate-limit error comes back instead of swallowing it.

### Login

#### [MAJOR] No client-side required-field validation on login/register/forgot forms — empty submit sends empty strings to backend

- **Flow step:** Login
- **File:** `index.html:939-952` (`auth-login-email`, `auth-login-password` — no `required`
  attribute); same pattern on `auth-reg-email` (:975), `auth-forgot-email` (:1036)
- **Repro:** Open login panel, submit with both fields empty → confirmed `POST /api/v1/auth/login`
  fires with body `{"email":"","password":""}`. The only reason an error appears at all is the
  backend's own validation message being relayed verbatim.
- **Impact:** Wastes a network round-trip on every accidental empty submit; the FE has zero
  opinion on obviously-invalid input and is fully dependent on the backend returning a sensible
  message. If the backend ever returns a generic 500 or unclear error for this case, the user sees
  a confusing/technical message instead of "Please enter your email and password."
- **Fix direction:** Add `required` to all auth text/password/email inputs, or check for empty
  values before calling `api.auth.*`.

#### [MINOR] Login error text not associated with its field via aria-describedby

- **Flow step:** Login
- **File:** `index.html:948-953` (`auth-login-password` input, `auth-login-error` paragraph) — no
  `aria-describedby` linking them; `#auth-login-error` also has no `role="alert"`. Same pattern
  confirmed on `auth-reg-error`, `auth-forgot-error`, `auth-reset-error` — all plain `<p>` with
  `hidden`, no `role`/`aria-live`.
- **Repro:** Trigger a 401 login error → confirmed `aria-describedby` attribute is `null` on the
  password input, and `#auth-login-error` has no `role` attribute (only `class="auth-error"`). A
  separate global `#a11y-announcer` live region exists in the app (used by
  `js/content/formatting.js`) but is never populated by `AuthModal._showError`.
- **Impact:** A screen reader user submitting the form has no automatic announcement that an error
  appeared, and no programmatic link between the error text and the field it concerns.
- **Fix direction:** Add `role="alert"` (or `aria-live="polite"`) to each `.auth-error` element,
  and `aria-describedby` on the related input(s) pointing to the error element's id.

#### [MINOR] Unverified-account login attempt shows identical "we just sent you a link" copy as a fresh registration, even though no new email was sent by that action

- **Flow step:** Login
- **File:** `js/auth.js:213-215` (`login()` catches 403 → `AuthModal._swap("verify")`);
  `index.html:1002-1008` verify panel copy: "We sent a verification link to your inbox. Click it
  to finish signing up."
- **Repro:** Mock `POST /auth/login` → 403 → confirmed FE swaps straight to `#auth-panel-verify`
  showing "We sent a verification link to your inbox" — but login doesn't call resend/register, so
  no new email was actually sent at that moment; this is just reusing the post-register panel.
- **Impact:** Misleading — implies a fresh email just landed in the user's inbox when actually
  they're seeing state left over from registration copy. A user who registered days ago and lost
  the original email may wait for a new one that was never sent, then get confused about why
  "resend" is needed if one was supposedly "just sent."
- **Fix direction:** Give the login-triggered verify panel distinct copy, e.g. "Your email isn't
  verified yet. Check your inbox, or resend the link below" — without implying an email was just
  dispatched.

Confirmed live and working correctly: wrong-password (401) shows "Incorrect email or password."
inline without clearing the typed password; Enter-key submit from the password field fires the
same login flow as clicking the button; already-registered email on the register panel shows a
clear "An account with this email already exists." error while keeping the "Log in" swap link
visible (no dead end).

### Session reflection (reload / multi-tab)

#### [MAJOR] Second open tab never reflects login/logout that happened in another tab without a manual reload

- **Flow step:** Session reflection (reload / multi-tab)
- **File:** `js/auth.js:176,212,290,327` — `wiki:session-changed` is a
  `document.dispatchEvent(new CustomEvent(...))`, same-document only;
  `js/storage/settings-theme.js:714` multi-tab sync listens on `window "storage"` events but only
  for bookmarks/recents/read-tracking `localStorage` keys — no key or `BroadcastChannel` exists
  for session/auth state.
- **Repro:** Two tabs (same context) both start logged out. Log in on Tab A → confirmed Tab A's
  topbar flips to "Logout" immediately. Tab B, left untouched with no interaction, still shows
  "Login" after a 500ms wait. Only after manually reloading Tab B does it pick up "Logout" (fresh
  `GET /auth/me`). Symmetric case: logging out on Tab A while Tab B stays open also leaves Tab B's
  topbar stale (still shows the pre-logout state) with no interaction.
- **Impact:** A user logged in on one tab who has (or opens) a second tab sees inconsistent, stale
  auth state until that tab happens to make an API call that 401s or is manually refreshed. Could
  be more than cosmetic if a stale tab lets a user attempt an action assuming the opposite auth
  state from reality.
- **Fix direction:** Broadcast session changes cross-tab — a `BroadcastChannel("wiki-session")`
  posted alongside the existing `wiki:session-changed` CustomEvent, or a dedicated `localStorage`
  key bumped on login/logout so the existing `window "storage"` listener can pick it up and
  trigger `Auth.refreshButtons()` / a re-render in other tabs.

Confirmed live and working correctly: session survives a same-tab reload (topbar stays "Logout"
after `page.reload()`), driven by the boot-time `GET /auth/me` in `Auth.init()`.

### Logout

No findings. Confirmed live: clicking the topbar button while logged in triggers `Auth.logout()`
directly (no confirmation dialog, which is reasonable for a low-risk action), topbar correctly
flips back to "Login", and pressing the browser Back button afterward does not resurrect any
stale authenticated UI (topbar stays "Login" after `page.go_back()`).

### Forgot / reset password

#### [MAJOR] Reset-password panel is a dead end on failure — no link back to login or to request a new reset email

- **Flow step:** Forgot / reset password
- **File:** `index.html:1054-1074` `#auth-panel-reset` — confirmed via DOM query this panel has
  zero `.auth-swap` elements, unlike every other panel (login has 2, register has 1, verify has 1,
  forgot has 1).
- **Repro:** Navigate to `/?mode=reset&token=expiredtoken` with backend mocked to return an
  expired-token error on submit → error text "This reset link has expired." shows correctly, but
  the panel offers no button/link to go back to login or re-request a reset email. Confirmed only
  escape hatches are the modal's X button, backdrop click, or Escape key — none of which explain
  what to do next.
- **Impact:** A user whose reset link expired (a very normal case — they waited too long, or
  clicked an old email) hits a genuine dead end: they must know to close the modal themselves and
  manually click "Forgot password?" again from the login panel. No guidance is given in the UI.
- **Fix direction:** Add a swap link/button to `#auth-panel-reset`, e.g. "Request a new reset
  link" → `AuthModal._swap("forgot")`, especially conditioned on/always visible after an error.

Confirmed live and working correctly: forgot-password messaging is genuinely enumeration-safe — a
known-registered email and an unregistered email both produce identical FE copy ("If that account
exists, a reset link is on its way."), confirmed byte-for-byte identical in both cases (mocked
backend returning 200 either way, consistent with the FE's blind trust of that response). Reset
success flow logs the user in and closes the modal correctly. Double-submit on both
forgot-password and reset-password confirmed (2× requests each on rapid double-click) — tracked
under the systemic double-submit finding in Register form.

### Keyboard & screen-reader

#### [CONFIRMED][MAJOR] No focus trap in auth modal — Tab moves focus into the hidden/background page while modal is open

- **Flow step:** Keyboard & screen-reader
- **File:** `js/auth.js` `AuthModal.open`/`_swap` — no keydown Tab handler anywhere;
  `index.html:915-1076`
- **Repro:** Open login panel, press Tab repeatedly and log `document.activeElement` each time.
  Confirmed trace: focus correctly cycles
  `auth-login-email → auth-login-password → auth-login-submit → auth-to-forgot → auth-to-register`,
  then on the next Tab it lands on `<body>`, then an `<a>` and `<button>` behind the modal, then
  `#auth-btn-home` itself (the very button that opened the modal, now hidden behind the backdrop),
  then `#scroll-top` and other background topbar controls, before eventually cycling back into the
  modal via `#auth-close`.
- **Impact:** A keyboard-only or screen-reader user tabbing through the form can silently leave
  the modal and start interacting with background page content that is supposed to be inert while
  the modal is open (the modal has no `inert`/`aria-hidden` applied to the rest of the page
  either). This also means a sighted keyboard user can lose track of their focus location, since
  the modal remains visually open while focus moves to invisible-behind-backdrop elements.
- **Fix direction:** Add a Tab/Shift+Tab keydown handler on the modal that cycles focus between the
  first and last focusable elements inside `.auth-dialog` (standard focus-trap pattern), and/or
  set `inert` on the rest of `#app`/body content while the modal is open.

**[MINOR] Login error text not associated with its field via aria-describedby** — see full entry
under Login; same finding, cross-referenced here since it's also a keyboard/screen-reader concern
(not counted separately in the severity tally).

Confirmed live and working correctly: Escape key closes the auth modal (via the app-wide keydown
handler in `app.js`); Enter-key submit works from the login password field; initial focus on
modal open correctly lands on the first field of each panel (`_swap`'s `focusId` map).

### Mobile (bottom sheet, 390px)

#### [CONFIRMED][MINOR] Drag handle on mobile bottom sheet is purely decorative — swiping it does not dismiss the sheet

- **Flow step:** Mobile (bottom sheet, 390px)
- **File:** `index.html:925` `.auth-drag-handle`; `css/components-auth.css:146-154` (visual only);
  no touch/pointer event listeners exist anywhere in `js/app.js` or `js/app/*.js` tied to
  `.auth-drag-handle`.
- **Repro:** At 390px viewport, open the auth modal, perform a mouse-down + drag-down + mouse-up
  gesture on the drag handle (simulating a swipe) → confirmed modal class stays `auth-modal`
  (still open), no dismissal occurs.
- **Impact:** The handle is a strong, standard visual affordance (used across iOS/Android/web
  bottom sheets) implying swipe-to-dismiss; here it does nothing. Users who try to swipe it away
  will be confused when it doesn't respond, though tap-outside/Escape/X still work as alternatives.
- **Fix direction:** Either wire up a swipe-down-to-dismiss gesture on the handle, or remove the
  handle to avoid the false affordance.

#### [MINOR] Swap links (Forgot password?, Register, Log in, etc.) have ~22px touch targets on mobile, below the 44px minimum

- **Flow step:** Mobile (bottom sheet, 390px)
- **File:** `css/components-auth.css:118-123` `.auth-link` / `.auth-swap` — no explicit
  padding/min-height set on the link buttons themselves.
- **Repro:** At 390px, measured bounding boxes for `#auth-forgot-to-login`, `#auth-to-register`,
  and the parent `<p class="auth-swap">` — all consistently ~21.75px tall with no extra hit-area
  padding from the parent. Compare to `#auth-close` which is a proper 44×44px target.
- **Impact:** Below Apple HIG (44pt) and WCAG 2.5.5 (44×44 CSS px) minimum touch target guidance;
  on a real device these small text links (e.g. "Forgot password?", "Register") are easy to
  mis-tap, especially inside a bottom sheet reachable one-handed near the thumb.
- **Fix direction:** Add vertical padding to `.auth-link`/`.auth-swap` to reach at least a 44px
  effective tap height, or increase line-height/padding on the `<p class="auth-swap">` wrapper.

Confirmed live and working correctly: bottom sheet correctly anchors to the viewport bottom after
its slide-up animation settles at 390px; the register password checklist stays within the dialog
bounds; on a heavily-shrunk viewport (390×500, simulating an on-screen keyboard covering the lower
half of the screen), the submit button is pushed below the fold but the dialog's
`overflow-y: auto` correctly allows scrolling to reach it — not a dead end, just requires a scroll.
