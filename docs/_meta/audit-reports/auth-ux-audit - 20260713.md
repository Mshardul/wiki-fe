# Auth UX Audit

Generated by auth UX audit agent. FE-only — see wiki-be's companion audit for backend findings.

**Methodology note:** Phase 2 (live browser verification via Playwright MCP) could not run in this
environment — the Playwright MCP server is hard-configured to launch the system "chrome" channel,
which is not installed on this machine, and installing it requires `sudo` (unavailable, and out of
scope for a no-node/build-free repo anyway). This is an environment limitation, not a decision to
skip verification. All findings below are therefore **static code-read findings** from `js/auth.js`,
`js/api.js`, `css/components-auth.css`, `index.html`'s auth modal markup, `js/app.js` boot/event
wiring, and `js/storage/settings-theme.js`'s `Sync`. Findings are marked `[CODE-CONFIRMED]` where
the bug is directly readable in source (e.g. a missing reset, a swallowed catch), or
`[UNVERIFIED — needs live browser]` where the concern depends on runtime/browser behavior this pass
could not observe (actual autofill `input`-event firing, actual focus behavior, actual timing/paint,
actual keyboard-obscuring-viewport behavior on a real mobile device). Items in the latter category
are still included because they are well-founded from the code, but should be re-verified live
before being treated as ground truth.

## Findings by flow step

### Discovery

### Register form

#### [MAJOR] Switching away from register panel and back leaves stale password input + desynced submit-disabled state

- **Flow step:** Register form — panel swap
- **File:** `js/auth.js:140-160` (`AuthModal._swap`), `js/auth.js:401-408` (`_wireModalInputs`)
- **Repro:** Open register panel → type a password that satisfies all rules (checklist turns green,
  `#auth-reg-submit` becomes enabled by the `input` listener) → click "Log in" to swap to the login
  panel → click "Register" to swap back. `_swap()` calls
  `this._renderChecklist("auth-pw-checklist", "")`, which force-renders the checklist to all-red
  using an **empty string**, but it never touches `#auth-reg-password`'s actual `.value` (still
  holds the previously typed password — inputs aren't reset on panel swap) and never touches
  `#auth-reg-submit.disabled` (last set by the `input` listener, not by `_swap`). Net result: the
  checklist visually shows all rules failing (red) while the actual input still contains a fully
  valid password and the submit button may still be sitting enabled from before — checklist and
  submit-gate are now out of sync with each other and with the field's real content.
- **Impact:** User sees a red/failing checklist for a password they already typed correctly, which
  reads as "my password is invalid" (confusing, makes them re-type unnecessarily) — or, in the
  inverse timing, the submit button stays enabled while the checklist shows all-red, so a submit
  could fire with a checklist that visually disagrees with what's about to be sent. Same pattern
  exists for the reset-password panel (`auth-reset-pw-checklist` / `auth-reset-submit`).
- **Fix direction:** in `_swap`, either clear the actual input value on leaving/entering
  register/reset panels, or re-derive both the checklist and the submit-disabled state from the
  input's current `.value` (call the same logic the `input` listener uses) instead of hardcoding an
  empty-string re-render.

#### [MINOR] No confirm-password field on register

- **Flow step:** Register form
- **File:** `index.html:1181-1218`
- **Repro:** Only a single masked password `<input>` exists in the register panel — no "confirm
  password" field, and no show/hide-password toggle either.
- **Impact:** A typo in a required 12+ character password with mixed case/digit/symbol is easy to
  make and invisible; the user only discovers it on the next failed login. Not necessarily wrong
  (some products deliberately skip confirm-password in favor of a reveal toggle) but this app has
  neither safety net.
- **Fix direction:** add a show/hide toggle button on the password field (cheap, no schema change)
  as the minimum mitigation if confirm-password is intentionally out of scope.

#### [MINOR] No loading indicator beyond a dimmed button; fields stay editable mid-request

- **Flow step:** Register form (also applies to login/forgot/reset — see below)
- **File:** `js/auth.js:193-202` (`_withSubmitGuard`), `css/components-auth.css:114-117`
  (`.auth-submit:disabled`)
- **Repro:** Submit register form → `#auth-reg-submit` gets `disabled` (opacity 0.5, cursor
  not-allowed) but its label text never changes ("Create account" stays "Create account" — no
  "Creating…" state, no spinner) and the email/password `<input>` fields are never disabled, so the
  user can keep editing them while the request is in flight.
- **Impact:** With a 15s client timeout (`api.js` `_DEFAULT_TIMEOUT_MS`), a slow network leaves the
  user looking at a slightly-dimmed button with no other feedback for up to 15 seconds — easy to
  read as "did my click even register?" and re-click (blocked by the guard, but nothing tells the
  user that). Editable fields during a pending request also mean a user could change the email
  they're about to see an error for, without realizing which email the in-flight request used.
- **Fix direction:** swap button label to a "Creating account…" state and/or add a spinner; disable
  the form's inputs (or at least visually indicate they're locked) during the request.

#### [MINOR] Double-submit is well-guarded — noting as a confirmed non-issue

- **Flow step:** Register form / all auth forms
- **File:** `js/auth.js:193-202` (`_withSubmitGuard`)
- Rapid double-click or Enter-then-click cannot fire two requests: `btn.disabled = true` is set
  synchronously before any `await`, and JS's single-threaded execution means the second invocation's
  `if (btn?.disabled) return;` check always sees the already-disabled button. This is solid and
  worth calling out as good, not a finding to fix — logged here only so the audit shows it was
  checked, per the journey checklist's explicit ask about double-submit races.

### Email verification

#### [MAJOR] [CODE-CONFIRMED] Verify-result panel's `<h2>` never updates — stays "Verifying your email" forever

- **Flow step:** Email verification — following the emailed link
- **File:** `js/auth.js:319-333` (`verifyFromLink`), `index.html:1236-1246`
  (`#auth-panel-verify-result`)
- **Repro:** User clicks the verification link in their email, landing on
  `?mode=verify&token=...`. `handleBootParams()` calls `verifyFromLink(token)`, which opens the
  `verify-result` panel and updates only `#auth-verify-result-copy` (the body paragraph) — to
  "Email verified! You can log in now." on success, or the error message on failure. The panel's
  `<h2 class="auth-title">Verifying your email</h2>` (`index.html:1237`) has **no `id`** and is
  never referenced or updated anywhere in `auth.js`. It is static markup only.
- **Impact:** Every outcome — success, expired token, malformed token, already-used token — leaves
  the modal's heading permanently reading "Verifying your email", while the paragraph below it says
  the process is done. This reads as stuck-mid-verification even when it succeeded or definitively
  failed; a user glancing only at the heading (common pattern) may think the page is hung and reload
  or navigate away before reading the paragraph, or before the "Back to log in" button
  (`#auth-verify-result-to-login`, `hidden` by default and only un-hidden after the try/catch
  settles) has appeared.
- **Fix direction:** give the `<h2>` an id and set it alongside the copy paragraph in both the
  success and catch branches of `verifyFromLink` (e.g. "Email verified" / "Verification failed").

#### [MINOR] Expired/malformed/already-used token all funnel through one generic error path

- **Flow step:** Email verification — error cases
- **File:** `js/auth.js:319-333` (`verifyFromLink`)
- **Repro:** The `catch` block sets the copy to `e.message` if `e instanceof ApiError`, else the
  hardcoded fallback "This link is invalid or has expired." There is no branching on `e.code` to
  differentiate an expired token from a malformed one from an already-used one — whatever message
  the backend's `ApiError` carries is shown verbatim, or the same generic fallback for anything else
  (including a network/timeout failure, which would otherwise get a technical message — see the
  Copy & tone section for that specific leak).
- **Impact:** If the backend does distinguish these cases with different `code`s (plausible per
  `auth.md`'s typed-error contract), the FE doesn't take advantage of it to tell the user what to do
  next differently (e.g. "already used — you're verified, log in" vs. "expired — resend a new one",
  which would ideally offer a resend action right there, but this panel has no resend button at
  all — only "Back to log in", hidden until settled).
- **Fix direction:** switch on `e.code` and, at minimum, surface a "Resend verification email" link
  on this panel rather than sending the user back to login with no direct path to a fresh link.

#### [MINOR] No resend affordance on the verify-result (post-link-click) panel

- **Flow step:** Email verification
- **File:** `index.html:1236-1246`
- **Repro:** The only action on this panel is "Back to log in" (and even that's hidden until the
  async verify call settles). If the token is expired, the user's only path to a new email is: go
  back to login → try to log in (will fail, account unverified) → get redirected to the verify-pending
  panel via the 403 branch in `login()` → click resend. That's an indirect, multi-step detour for
  what should be a one-click "send me a new link" from the error screen itself.
- **Fix direction:** add a resend button directly on the verify-result panel when the outcome is a
  token error.

### Resend verification

#### [MAJOR] Resend always reports success, even on genuine failure — no differentiated or rate-limit messaging

- **Flow step:** Resend verification
- **File:** `js/auth.js:309-317`
- **Repro:**
  ```js
  async resend(email) {
    try {
      await api.auth.resend(email);
      showToast("Verification email sent");
    } catch {
      /* generic 200 either way; still confirm so the click isn't silent */
      showToast("Verification email sent");
    }
  }
  ```
  Every outcome — success, rate-limited, network failure, backend down — shows the identical
  "Verification email sent" toast. The comment shows this is deliberate (likely to avoid leaking
  whether an email is registered), but it's applied indiscriminately to *all* error types, including
  ones that have nothing to do with account enumeration (network failure, request timeout, backend
  5xx).
- **Impact:** If a user is rate-limited by the backend after clicking resend a few times, or the
  request times out, they are told it succeeded and will wait for an email that never arrives, with
  zero signal that anything went wrong. There's no way to distinguish "check your spam folder" from
  "actually broken right now, try again in a minute."
- **Fix direction:** keep the uniform-success messaging for auth-domain error codes (to preserve
  anti-enumeration), but let `NETWORK`/`TIMEOUT` `ApiError`s (which reveal nothing about account
  existence) surface a distinct, honest message like "Couldn't reach the server — check your
  connection and try again."

#### [MINOR] Resend button has no disabled/cooldown state between clicks

- **Flow step:** Resend verification
- **File:** `js/auth.js:428-434` (wiring), `index.html:1227`
- **Repro:** `#auth-resend-btn` is wrapped in `_withSubmitGuard`, so a single rapid double-click
  can't double-fire (per the earlier double-submit finding), but nothing stops the user from
  clicking it again 2 seconds later, then again, then again — no cooldown/rate-limit UI on the FE
  side at all. Combined with the always-success toast above, a user can hammer this button and get
  "Verification email sent" every time with no indication they're now being throttled server-side.
- **Fix direction:** add a client-side cooldown (e.g. disable + "Resend in 30s" countdown) after a
  successful send, independent of whatever the backend enforces.

### Login

#### [MAJOR] Network/timeout errors leak raw technical text into user-facing error messages

- **Flow step:** Login (also register, forgot-password, reset-password — same pattern, filed once
  here since it's one root cause)
- **File:** `js/auth.js:276-281` (login catch), `js/api.js:34-38`
- **Repro:** `api.js`'s `_request` throws `ApiError("NETWORK", networkErr.message, 0)` for any
  non-abort fetch failure — `networkErr.message` is the raw browser string (e.g. `"Failed to
  fetch"` or `"NetworkError when attempting to fetch resource."` depending on browser). Every
  catch block in `auth.js` does the same pattern:
  ```js
  AuthModal._showError("auth-login-error", e instanceof ApiError ? e.message : "Login failed");
  ```
  Since a `NETWORK` failure **is** an `ApiError` (just with a browser-generated message instead of a
  backend one), `e.message` is shown directly — meaning a dropped connection surfaces something like
  "Failed to fetch" in the login error `<p>`, verbatim.
- **Impact:** Directly violates this repo's own stated rule ("no raw technical/API-error text
  leaking to the user") and CONVENTIONS.md's "fail loud in dev, but surface through UI, not
  console" — this *does* surface through UI, but as a raw technical string rather than a
  human message. A real user has no idea what "Failed to fetch" means or what to do about it.
- **Fix direction:** in each catch block, branch on `e.code` first — `TIMEOUT` already has a
  purpose-written message from `api.js` ("Request timed out. Please try again.") which is fine to
  show as-is; `NETWORK` should get its own FE-authored message ("Couldn't reach the server. Check
  your connection and try again.") instead of falling through to `e.message`.

#### [MINOR] Unverified-account login (403) reuses the verify-pending panel's title/copy elements correctly, but silently discards the login form's typed password

- **Flow step:** Login — unverified account attempt
- **File:** `js/auth.js:255-283` (`login`)
- **Repro:** Logging in with correct credentials for an unverified account gets a 403, swaps to the
  `verify` panel, and sets title/copy to "Account not verified" / "resend the verification email"
  copy. This part works. However `resend()` (wired at `auth.js:428-434`) reads the email from
  `#auth-reg-email` (the *register* panel's email field), not from `#auth-login-email` (where the
  user actually just typed it):
  ```js
  document.getElementById("auth-resend-btn")?.addEventListener("click", () =>
    _withSubmitGuard("auth-resend-btn", () =>
      this.resend(document.getElementById("auth-reg-email").value.trim()),
    ),
  );
  ```
  If the user reached the verify panel via a failed *login* (not via register), `#auth-reg-email`
  was never populated in this session — `resend()` would fire with an empty string.
- **Impact:** Clicking "Resend email" after a failed unverified-login attempt likely calls
  `api.auth.resend("")` — an empty email — rather than the account the user is actually trying to
  verify. Given `resend()` always shows "Verification email sent" regardless of outcome (see above),
  the user has no way to notice this is silently going to the wrong (empty) address.
  This is a **confirmed code-level bug**, not a hypothesis: the DOM id read is unconditionally
  `auth-reg-email` with no branch for "we got here via login, not register."
- **Fix direction:** track which email triggered the verify-panel swap (from either the login or
  register flow) in a small piece of state, and have `resend()` read from that instead of hardcoding
  the register panel's field.

#### [UNVERIFIED — needs live browser] Password checklist may not re-validate on browser autofill

- **Flow step:** Login / Register — browser password-manager autofill
- **File:** `js/auth.js:403-408` (checklist wiring, register), `js/auth.js:451-456` (reset)
- **Repro (code-level):** The checklist and submit-enable logic are wired exclusively to the native
  `input` event: `pw?.addEventListener("input", () => { ... })`. Whether a browser's built-in
  autofill (not a password manager extension, the browser's own saved-credentials autofill) fires a
  real `input` event on the target field is browser/version-dependent — modern Chrome generally
  does, but this cannot be confirmed without driving an actual autofill in a real browser session,
  which Phase 2 couldn't do here (see methodology note at top). If it does *not* fire in whatever
  the target browsers are, the register submit button would stay `disabled` (its initial HTML state
  is `disabled` — `index.html:1208`) even though the autofilled password is fully valid, and the
  checklist would stay all-red, since nothing else ever calls `_renderChecklist` or re-evaluates
  `validatePassword` after initial render.
- **Impact if true:** a user whose browser autofills the register form would see a permanently
  disabled "Create account" button with an all-red checklist, with no visible reason why, since the
  password looks correct to the eye (dots/masked).
- **Fix direction:** re-verify live; if confirmed, add a fallback re-check on `focus`/`blur` of the
  password field, or poll/observe the field's value shortly after the form/panel becomes visible.

#### [MINOR] No visible "wrong password" vs "unknown email" distinction — worth confirming intentional

- **Flow step:** Login — error cases
- **File:** `js/auth.js:266-282`
- **Repro:** All non-403 login failures funnel to the same `auth-login-error` element with
  `e.message` from the backend. Since the FE can't distinguish these (nor should it, if the backend
  intentionally returns one generic "invalid credentials" message for both — good anti-enumeration

### Session reflection (reload / multi-tab)

#### [POLISH] Confirmed-good: reload persists session, second tab syncs via storage event

- **Flow step:** Session reflection
- **File:** `js/auth.js:222-234` (`Auth.init`), `js/auth.js:484-498` (storage listener)
- On boot, `Auth.init()` calls `GET /auth/me` once and sets `state.session` — since identity is
  never cached in localStorage (by design, per CONVENTIONS.md), a reload correctly re-derives
  session state from the httpOnly-cookie-backed endpoint rather than trusting stale local data. A
  second tab picks up login/logout via the `wiki-session-sync` localStorage bump
  (`_broadcastSessionChange`), which fires a native `storage` event in *other* tabs only (same-tab
  writes don't self-fire, correctly documented in a comment) — that listener re-probes `/auth/me`
  and re-renders. This is a sound design; no bug found in the read. Noting as checked, not as an
  issue, since the journey checklist explicitly calls out multi-tab as something to verify.

#### [MINOR] No gating for protected content — confirmed this app has none, not a bug

- **Flow step:** Session reflection / gating
- **File:** `js/storage/{bookmarks,recents,read-tracking}.js`, `js/storage/settings-theme.js`
- The only consumers of `state.session.status === "in"` are the four `storage/` modules deciding
  whether to sync a given local write to the backend — there is no concept of "protected" wiki
  content gated behind login anywhere in the codebase; all articles are public regardless of auth
  state. This matches the product model (personal layer = bookmarks/recents/reads sync, not content
  access control). Logged as checked/non-issue per the journey checklist's explicit ask.

### Logout

#### [MINOR] Logout has no user-facing confirmation or loading state

- **Flow step:** Logout
- **File:** `js/auth.js:390-399` (`logout`), `js/app.js:233-235` (`auth-toggle` wiring via
  `Auth.toggle()`)
- **Repro:** Clicking the topbar auth button while logged in calls `Auth.toggle()` →
  `this.logout()` directly — no confirmation dialog (reasonable for logout, most apps skip this) but
  also no toast/feedback of any kind once it completes. The button label flips from "Logout" to
  "Login" via `refreshButtons()`, which is a signal, but it's the only one — logout is
  fire-and-forget with `Sync.flushBestEffort().catch(() => {})` and `api.auth.logout().catch(() =>
  {})`, both errors silently swallowed.
- **Impact:** If `api.auth.logout()` fails server-side (network blip), the FE still proceeds to
  clear local session state and show "Login" — the user believes they're logged out client-side
  (correct, `state.session` is cleared) but the server-side session/cookie may still be technically
  live if that specific call failed to reach the backend. This is a defensible "B-lite" trade-off
  (comment explicitly calls it out: "best-effort flush, then clear + logout regardless of result")
  but it means there's no path for the user to ever learn a logout call failed server-side.
- **Fix direction:** likely fine as-is given the explicit design comment, but worth a toast on the
  rare logout-API-failure path for transparency ("Logged out on this device" vs silence).

#### [UNVERIFIED — needs live browser] Browser back-button after logout — stale authenticated UI risk

- **Flow step:** Logout — browser back navigation
- **File:** N/A — depends on browser bfcache behavior, not directly controllable from a static read
- **Note:** `js/app.js` does have a `pageshow` handler that re-routes on `e.persisted` (bfcache
  restore), which is a good sign this was considered for *content* navigation — but that handler
  calls `route()`, not `Auth.init()` or `Auth.refreshButtons()`, so a bfcache-restored page after

  this is exactly the kind of thing that needs an actual back-button press to confirm.

### Forgot / reset password

#### [MAJOR] `#auth-forgot-sent` success message can reappear stale on a later, still-pending submission

- **Flow step:** Forgot password
- **File:** `js/auth.js:335-345` (`forgotPassword`), `js/auth.js:169-179` (`_clearErrors`),
  `index.html:1262-1264`
- **Repro:** `forgotPassword()` sets `document.getElementById("auth-forgot-sent").hidden = false` on
  success. Nothing ever sets it back to `hidden = true` except the initial HTML state — `_swap()`
  calls `_clearErrors()` on every panel change, but `_clearErrors()` only touches the four
  `*-error` elements (`auth-login-error`, `auth-reg-error`, `auth-forgot-error`,
  `auth-reset-error`); `auth-forgot-sent` is not in that list.
- **Repro walkthrough:** Request a reset for `a@x.com` → success message shows → close modal →
  reopen → go to forgot panel → the stale "If that account exists, a reset link is on its way."
  message is still showing (never got hidden again) → user types `b@x.com` and hasn't submitted
  yet → the success message is already visible before they've done anything this time, which reads
  as if the page preemptively confirmed success for a request that hasn't been made.
- **Impact:** Confusing/misleading state — a success message displayed before (or independent of)
  the action that's supposed to produce it undermines trust in the whole form's feedback.
- **Fix direction:** add `auth-forgot-sent` to `_clearErrors()`'s reset list (or a dedicated
  panel-enter reset), and/or hide it again at the start of every `forgotPassword()` call before the
  request resolves.

#### [SECURITY] [POLISH] Forgot-password success copy is good anti-enumeration practice — confirmed correct, not a finding

- **Flow step:** Forgot password
- **File:** `index.html:1262-1264`, `js/auth.js:335-345`
- The success copy "If that account exists, a reset link is on its way." is deliberately

  (`auth-forgot-error`, shown only if the request itself throws, e.g. network failure) is separate
  from the "not found" case, which per this copy is presumably folded into the same success message
  server-side (consistent 200-style response) rather than surfaced as an FE error — this looks
  right from the FE's vantage point. Logging as checked/confirmed-good per the journey checklist's
  explicit ask about enumeration-safe messaging; the actual backend behavior (same latency, same
  response shape) can't be confirmed without treating wiki-be as non-black-box, which is out of
  scope.

- Same finding as the register-form entry — one masked password field, no confirm, no reveal

#### [UNVERIFIED — needs live browser / BE] Whether other active sessions get invalidated on password reset

- **Flow step:** Reset password

### Anon → account migration

#### [MAJOR] [CODE-CONFIRMED] Failed migration import is completely silent — user believes data was kept when it wasn't

- **Flow step:** Anon → account migration
- **File:** `js/auth.js:52-87` (`maybeMigrate`), specifically line 83:
  `await api.importAll(payload).catch(() => {});`
- **Repro:** User has anon bookmarks/recents/reads → logs in → toast prompts "You have unsaved items
  on this device from browsing signed out. Keep them in your account?" → user clicks "Keep them" →
  `api.importAll(payload)` is called and its failure is swallowed with an empty catch — no toast, no
  retry, no error surfaced anywhere. Immediately after (success or failure, no branch on it),
  `login()`'s caller continues to `await Sync.pullAll()`, which **overwrites local
  storage with server truth** (`settings-theme.js:828-861`). If the import failed, server truth has
  none of the migrated data, so `pullAll()` then wipes the local anon data that was still sitting in
  localStorage right after the user explicitly chose to keep it.
- **Impact:** This is the single most damaging finding in the audit from a trust standpoint — a user
  can explicitly opt to preserve their data, be shown no error, and have it silently deleted by the
  very next line of code. There's no recovery path once `pullAll()` has run.
- **Fix direction:** don't swallow the `importAll` error — on failure, show an error toast ("Couldn't
  save your local data — it's still on this device, try again from your account") and **skip** the
  subsequent `Sync.pullAll()` overwrite (or at minimum don't clear/overwrite the specific local keys
  that hold the not-yet-migrated data) so the user's local copy survives to retry.

#### [MINOR] "Discard" is implicit (toast timeout), not an explicit user action

- **Flow step:** Anon → account migration
- **File:** `js/auth.js:52-87`
- **Repro:** The only affirmative action is "Keep them." Not clicking it — whether because the user
  dismissed the toast, ignored it, or the 12.5s timer simply elapsed while they were reading the
  page — is treated identically as "Discard," which calls `Sync.clearUserDataCache()`. There's no
  way to distinguish "I actively chose not to keep this" from "I didn't notice a toast in the
  bottom-right corner in time."
- **Impact:** A 12-12.5 second window to notice, read, and act on a toast is short, especially for a
  message this consequential (it's about to discard local browsing history/bookmarks). Given the
  MAJOR finding above about migration silently failing even when the user *does* click "Keep them,"
  this compounds risk: the safe path (do nothing) and the intended-safe path (click Keep) can both
  end in data loss with zero error feedback.
- **Fix direction:** consider a non-expiring or longer-lived prompt for this specific
  action (this is a one-time, data-loss-risking decision, not a routine dismissible toast), or a
  persistent banner instead of a transient toast.

#### [MINOR] Toast internal timing has a ~500ms dead zone between visual dismissal and promise resolution — likely harmless but worth noting

- **Flow step:** Anon → account migration
- **File:** `js/auth.js:55-75`
- The toast is shown with `durationMs: 12000` (`render/toast.js` auto-advances/hides the toast
  element at that mark) while the wrapping `Promise` in `maybeMigrate` has its own independent
  `setTimeout(() => done(false), 12500)`. Nothing calls `done(false)` when the toast itself expires
  at 12000ms — only the "Keep them" button click or the 12500ms fallback resolve the promise. This
  means for roughly 500ms after the toast visually disappears, the `await` in `maybeMigrate` is
  still pending (delaying `Sync.pullAll()` from starting by up to 500ms with no user-visible
  effect). Not a functional bug, just an odd design coupling between two independently-configured
  durations that happen to almost-but-not-quite match; a future edit to either duration could
  desync them further with no test coverage watching for it.
- **Fix direction:** derive one from the other (e.g. resolve the promise directly from the toast's
  own dismissal callback rather than a second independent timer) so there's a single source of
  truth for the window length.

- **Flow step:** Keyboard & screen-reader

#### [MINOR] No `aria-invalid` on inputs when their associated error is showing

- **Flow step:** Keyboard & screen-reader
- **File:** `js/auth.js` (all `_showError` call sites), `index.html` auth inputs
- **Repro:** `aria-describedby` correctly points each input at its error paragraph, so a screen
  reader *will* announce the error text when it appears (via the live region) and when the field is
  focused/described — but the input itself never gets `aria-invalid="true"` set to programmatically
  mark it as invalid, which is the more standard/robust pattern for assistive tech to flag "this
  specific field has a problem" (versus relying solely on the live-region announcement firing at
  the right moment, which a user tabbing back into the form later would miss).
- **Fix direction:** in `AuthModal._showError`, also set `aria-invalid="true"` on the related
  input, and clear it in `_clearErrors`.

#### [MINOR] Password checklist `<li>` items rely on a `::before` content glyph (✗/✓) for state — check color-only signaling

- **Flow step:** Keyboard & screen-reader
- **File:** `css/components-auth.css:89-97`
- The checklist correctly uses both a glyph change (✗ → ✓) *and* a color change (red → accent) —
  not color alone — which is good for colorblind users reading visually. However, the glyph and
  color are both delivered via CSS `::before` pseudo-element content, which is **not** exposed to
  screen readers at all (`::before`/`::after` generated content is inconsistently or not announced
  across screen reader/browser combinations, and is generally advised against for conveying
  meaningful state). The `<li>` text itself ("At least 12 characters" etc.) doesn't change between
  ok/not-ok states — only the class toggles (`class="ok"` or not), which drives the pseudo-element.
  A screen reader user gets the rule *label* read out but no reliable indication of whether it
  passed or failed, beyond the list having `aria-live="polite"` (which announces the list changing,
  not which items are which state).
- **Fix direction:** add visually-hidden text ("— met" / "— not met") inside each `<li>` alongside
  the visual glyph, or an `aria-label`/`aria-checked` pattern if converting the list to a proper
  status-list role.

#### [MINOR] Inconsistent verb: "Log in" (button/title) vs "sign in"/"signing out"/"signing up" used interchangeably in body copy

- **Flow step:** Copy & tone
- **File:** `index.html`, `js/auth.js`
- **Repro:** UI chrome (titles, buttons, breadcrumb) consistently says "Log in" / "Logout" — good.
  But body copy drifts: "to finish **signing up**" (verify-pending copy, twice), "from browsing
  **signed out**" (migration toast), while the actual button/title language is "Log in" /
  "Register" / "Create account" — "sign up" is never the label of any control, yet appears twice in
  prose referring to the same action "Register"/"Create account" performs. Similarly "Account not
  verified" (title) vs. the concept is always otherwise called "verify"/"verification" — consistent
  there, just flagging the sign-up/register mismatch specifically.
- **Impact:** Minor terminology drift — a user who registers via "Create account" and later reads
  "finish signing up" has to map two different terms to the same action. Small, but the task brief
  explicitly asks for this kind of consistency check.
- **Fix direction:** standardize on one verb pair throughout — since the buttons already commit to
  "Log in" / "Create account", prose should say "finish creating your account" rather than "finish
  signing up," or rename the button to "Sign up" project-wide (bigger change). Cheaper fix: align
