# Mobile UX Audit

Generated by mobile UX audit agent. Viewports tested: 360px, 375px, 393px, 768px.

## Findings by page/component

### Home view

#### [MAJOR] Wiki card pin button invisible and undiscoverable on touch

- **Page/component:** Home view — wiki card grid
- **File:** `css/view-home.css:164-188` (`.wiki-card-pin-btn`), `js/render/home-index.js:85-89`
- **Viewport(s):** 360px, 375px, 393px, 768px (all)
- **Repro:** Load home view on any mobile viewport. Pin button is `opacity:0` by default, only becomes visible via `.wiki-card:hover` or `.pinned` state. Confirmed live: computed opacity is `0` and box is 28x28px at all four viewports.
- **Impact:** Touch devices have no `:hover` — an unpinned card's pin star is completely invisible (confirmed live: `opacity:0` computed style, but the element is still hit-testable/tappable at its 28x28px box — so a user tapping blindly in the top-right corner of a card *would* successfully pin it with zero visual feedback confirming what happened, beyond a full grid re-render). Users can't discover the pin feature at all unless they already know the button exists at that exact pixel location. Even if found, 28x28px is under the 44x44px minimum touch target.
- **Fix direction:** Add `@media (pointer: coarse)` or a `responsive.css` rule to force `opacity:1` on `.wiki-card-pin-btn` and bump to 44x44px, similar to the `.anchor-btn` pattern already used at `css/responsive.css:203-212`.

### Index view

#### [MAJOR] Index card swipe gestures (bookmark/read-toggle) have no visible affordance and no discoverability

- **Page/component:** Index view — index card grid
- **File:** `js/render/home-index.js:456-564` (`bindIndexCardSwipe`), `css/view-index.css:224-258`
- **Viewport(s):** 360px, 375px, 393px (touch-relevant; less relevant at 768px if used with mouse)
- **Repro:** Static read: swipe right on an index card bookmarks it, swipe left toggles read/unread. The only affordance (🔖/👁 icon fade-in) only appears mid-swipe via `::before`/`::after` on `.card-swiping`; there is no static hint, icon, label, or onboarding tooltip indicating this gesture exists before a user accidentally triggers it.
- **Impact:** A first-time mobile user has no way to discover this feature exists. Worse, an accidental horizontal drag (e.g. imprecise tap-and-drag) silently bookmarks or marks-read/unread an article with only a toast as feedback — a surprising, easily-triggered side effect for an unfamiliar gesture.
- **Fix direction:** Add a one-time onboarding hint (e.g. first-run tooltip) or a persistent subtle visual cue (small icon in the corner) indicating cards are swipeable, especially since the gesture has real data-mutating side effects.

#### [MINOR] Index card swipe gesture can conflict with pull-to-refresh and back-navigation swipe

- **Page/component:** Index view — index card grid
- **File:** `js/render/home-index.js:456-624` (`bindIndexCardSwipe`, `bindIndexPullToRefresh`)
- **Viewport(s):** 360px, 375px, 393px
- **Repro:** Static read: both swipe handlers are bound to the same `#index-sections` container's touchstart/touchmove. Card swipe sets `state._cardSwipeActive = true` to "tell global edge-swipe (back-nav) to stand down" per the code comment, implying a known conflict was already mitigated for back-nav. - **Impact:** Potential gesture ambiguity for users starting a drag near the top of the index list — not confirmed broken live, but flagged as a hypothesis given three independent gesture systems (card swipe, pull-to-refresh, edge-swipe back) share the same touch surface.
- **Fix direction:** Cross-check with `js/app/mobile-panels.js` edge-swipe logic when auditing mobile panels (item 9) to confirm no real conflict.

### Content view — layout

- **Page/component:** Content view — layout / topbar
- **File:** `css/components/topbar.css` (`.topbar-icon-btn`, `.topbar-overflow-item`), `css/responsive.css:110-113` (44px fix only inside `max-width:640px`)
- **Viewport(s):** 768px
- **Repro:** Open any article at 768px viewport. - **Impact:** Every secondary topbar action (9 total icons at 768px) is a sub-44px touch target on a touch-capable tablet viewport. No overlap/crowding was found — the row fits (rightmost edge at 748px inside a 768px topbar) — but each icon individually is hard to hit precisely.
- **Fix direction:** Extend the 44px-minimum rule to `@media (pointer: coarse)` in addition to `max-width:640px`, matching the pattern already used for `.anchor-btn` and the `@media (pointer: coarse)` block at the end of `responsive.css`.

### Content view — TOC

#### [CRITICAL] TOC collapse chevron empties the mobile drawer, leaving a dead-end empty panel

- **Page/component:** Content view — TOC (mobile drawer)
- **File:** `js/content/toc.js:13-15` (`#toc-collapse` click handler, shared between desktop sidebar and mobile drawer), `css/view-content/layout.css:143-148` (`.toc-sidebar.collapsed .toc-nav{display:none}`)
- **Viewport(s):** 360px, 375px, 393px, 768px (all four — drawer markup is shared)
- **Repro:** Open any article on a mobile viewport → tap the TOC FAB to open the mobile drawer → tap the small chevron next to "ON THIS PAGE" (`#toc-collapse`, 22x22px). Confirmed live: this sets `.toc-sidebar.collapsed`, which sets `#toc-nav{display:none}` — the heading list disappears entirely, but the drawer itself stays open at full width/height, still covering most of the screen (280px wide, full viewport height), now showing only the "ON THIS PAGE" header and (if present) the Notes scratchpad below an empty gap. Screenshot confirms: user is left staring at a mostly-blank overlay panel.
- **Impact:** This collapse toggle is a legitimate desktop feature (shrink a persistent sidebar to save space while reading) but makes no sense inside a temporary, dismissible mobile drawer — the drawer's entire purpose is to show the TOC; collapsing it produces a dead-end UI state a user must know to reverse by tapping the same tiny 22x22px chevron again (no visible affordance suggests this), or close and reopen the drawer.
- **Fix direction:** Hide `#toc-collapse` inside `.toc-sidebar.mobile-open` (e.g. `.toc-sidebar.mobile-open .toc-collapse { display:none }`), since the drawer's own backdrop/swipe-down/FAB toggle already serves the "get it out of the way" function.

#### [CRITICAL — supersedes an earlier false-pass in this same audit] TOC drawer, its backdrop, and the TOC FAB are all mispositioned off-screen because `#view-content.active`'s permanent post-animation `transform` breaks `position:fixed` for every descendant

- **Page/component:** Content view — TOC (mobile drawer + FAB) — see also the cross-cutting root-cause writeup under "Responsive breakpoints (cross-cutting)", which this duplicates because the TOC drawer is the most visible symptom
- **File:** `css/base.css:167-173` (`.view.active { animation: viewEnter 0.25s ease forwards }`, no cleanup afterward), `index.html:289-651` (`#view-content` contains `.toc-mobile-btn`, `#toc-mobile-overlay`, `.toc-sidebar`, `#sticky-section-header`, `#article-find`, all `position:fixed`)
- **Viewport(s):** 360px (confirmed; root cause is viewport-independent)
- **Repro:** An earlier pass in this same audit run recorded the TOC drawer's backdrop-click/scroll-lock/edge-swipe as "Verified working" - that test happened to run under conditions where the bug's effect wasn't visible. Re-tested more rigorously: `getComputedStyle(document.getElementById('view-content')).transform` is `matrix(1,0,0,1,0,0)` (the identity matrix) for the entire time an article is open - a permanent artifact of `animation: viewEnter ... forwards` (and the nav-forward/nav-back variants), which all end on an explicit non-`none` `transform` and never get cleaned up. Per the CSS spec, *any* non-`none` `transform` on an ancestor - including the identity matrix - creates a new containing block for `position:fixed` descendants, so `.toc-mobile-btn`, `#toc-mobile-overlay`, `.toc-sidebar.mobile-open`, and `#sticky-section-header` are no longer fixed to the viewport; they're fixed relative to `#view-content`'s own (very tall, article-length) box instead. Confirmed live: opening the TOC drawer via `openMobileToc()` at `scrollY:0` on a fresh article load renders `#toc-sidebar` at `{y: -30428, ...}` - thousands of pixels off-screen, completely unusable. Tapping the on-screen coordinates where the TOC FAB visually should be hits article body text instead (`document.elementFromPoint` confirms this). `#scroll-top`, which is *not* a descendant of `#view-content` in the DOM, is unaffected and renders correctly, which is why it was visible in screenshots while the TOC FAB silently wasn't.
- **Impact:** This is the single highest-blast-radius bug in the audit: the primary mobile TOC entry point (FAB), the TOC drawer itself, its backdrop, and the sticky section header are all broken simultaneously, on every content-view page load, on every viewport, for any user without `prefers-reduced-motion` set. A mobile user effectively cannot open the TOC via the FAB at all in the common case.
- **Fix direction:** Remove the lingering `transform` after the view-enter animation completes - e.g. add `animation-fill-mode: none` once the animation ends (via a `transitionend`/`animationend` listener that clears the inline/computed transform), or restructure so `.view.active`'s animation only touches `opacity` (drop the `transform` from the keyframes, or apply the slide animation to an inner wrapper that fixed-position elements are NOT descendants of). Also audit `#article-find`, `.toc-sidebar` desktop-persistent mode, and any other `position:fixed` element nested inside `.view` for the same issue.

### Content view — mermaid diagrams

#### [CRITICAL] Mermaid diagram zoom overlay renders completely blank (0x0 SVG) — desktop and mobile

- **Page/component:** Content view — mermaid diagrams (zoom/lightbox)
- **File:** `js/content/zoom-lightbox.js:222-238` (`addDiagramZoom`), `css/view-content/mermaid.css:240-251` (`.zoom-overlay-content`, flex container with no defined height)
- **Viewport(s):** 360px, 768px, and confirmed also at 1200px desktop width — not mobile-specific, but explicitly in scope per this audit's checklist ("Diagrams are a classic mobile failure point")
- **Repro:** Open any article with a real Mermaid diagram (e.g. `#dsa/union-find`) → tap/click the diagram to zoom. The overlay opens correctly (backdrop dims, 44x44px close button appears, positioned correctly), but the diagram itself is entirely invisible. Confirmed live via DOM inspection: `addDiagramZoom` clones the rendered SVG and explicitly sets `width="100%" height="100%"` on the clone (`zoom-lightbox.js:232-233`) before appending it into `.zoom-overlay-content`, which is a flex container (`display:flex; align-items:center; justify-content:center`) with no explicit height of its own. `height:100%` on the SVG therefore resolves against a zero-height flex ancestor, collapsing the cloned SVG's rendered box to `0x0` (`getBoundingClientRect()` confirmed `{width:0, height:0}` at both 360px and 1200px viewports). Screenshot confirms: user taps a diagram expecting a full-screen zoomed view and gets a dimmed backdrop with only a close button — the diagram is gone.
- **Impact:** Diagram zoom — an explicitly-designed, carefully-built feature (pinch-zoom, double-tap-zoom, swipe-down-dismiss are all correctly implemented in the same file) — is completely non-functional for its core purpose. Any user trying to zoom into a Mermaid diagram to read small text sees nothing. This is the single most severe functional bug found in this audit.
- **Fix direction:** The clone's `width`/`height` should not be percentage-based against an undefined-height flex parent. Either give `.zoom-overlay-content` an explicit height (e.g. `height: 90vh` alongside its existing `max-height: 92vh`), or set the cloned SVG's dimensions from its own `viewBox`/original rendered size rather than `100%`, or add `min-height`/`flex: 1` handling. Also affects the desktop path — not gated to mobile at all, so likely a recent regression or an untested code path.
