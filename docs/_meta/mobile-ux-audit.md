# Mobile UX Audit

Generated by mobile UX audit agent. Viewports tested: 360px, 375px, 393px, 768px (+ landscape
variants 852x393/667x375 for orientation-change checks, plus a 393x250 keyboard-simulation
profile). Phase 1 (static code read) covered all 16 checklist items. Phase 2 (live browser
verification) started blocked by an MCP/Chromium-channel config issue, then completed after a
mid-session MCP reconnect using a real mobile-emulated Playwright context (`isMobile: true`,
`hasTouch: true`, iPhone UA) driven via `browser_run_code_unsafe` — no dedicated device-emulation
tool exists in this MCP tool set, so each check created and tore down its own context (no state
persists between calls in this runtime). Findings below are marked live-verified where a real
mobile browser context confirmed them; a few remain code-read-only where live verification would
require a physical notched device or real iOS WebKit (noted per-finding).

### Content view — layout

#### [MAJOR] Heading text breaks mid-word ("Table of Contents" → "Table of / Content / s") under OS text-zoom
- **Page/component:** Content view — layout (headings), cross-cutting text-zoom
- **File:** `css/view-content/layout.css:427-434` (`.markdown-body { overflow-wrap: break-word; word-break: break-word }`)
- **Viewport(s):** 360px, live-verified with `document.documentElement.style.fontSize = "200%"` simulating an OS text-size boost
- **Repro (live-verified):** Loaded the "Stack" article at 360px in a real mobile Playwright context, then set `document.documentElement.style.fontSize = "200%"` to simulate a 200% OS text-scale accessibility setting. Screenshot shows the "Table of Contents" H2 heading rendered across three lines as "Table of" / "Content" / "s" — `word-break: break-word` is splitting the word "Contents" mid-character once the heading's available inline width can no longer fit it at the enlarged font size, rather than wrapping at the word boundary before "Contents" only.
- **Impact:** Users who rely on OS-level text-size accessibility settings (common for low-vision users, exactly the population this kind of setting exists for) see broken mid-word hyphenation-without-a-hyphen on headings at narrow widths. It's readable but visually broken and looks like a rendering bug rather than an intentional accessibility accommodation.
- **Fix direction:** `word-break: break-word` is a blunt instrument better suited to long unbreakable strings (URLs, code) than natural-language headings. Consider `overflow-wrap: break-word` alone (already present) without `word-break: break-word` for heading elements specifically (`h1,h2,h3,h4`), since `overflow-wrap` already provides a same-effect fallback but prefers breaking at word boundaries first.

#### [MINOR] Mobile TOC drawer has no explicit safe-area-inset-bottom padding
- **Page/component:** Content view — TOC drawer (mobile)
- **File:** `css/view-content/layout.css:398-411` (`.toc-sidebar.mobile-open`)
- **Viewport(s):** all mobile viewports on notched/home-indicator iPhones (hypothesis — cannot be live-verified without a physical device; Playwright confirmed the padding value itself is unconditional but can't emulate real notch insets)
- **Repro:** `.toc-sidebar.mobile-open` is a `position:fixed` full-height (`top:0; bottom:0`) right-side drawer with `padding: var(--s6)` uniformly — no `env(safe-area-inset-bottom)` addition, unlike the wiki-switcher list, bookmarks modal, and auth dialog, which all explicitly pad for the bottom safe area in their mobile/bottom-sheet variants. Live-confirmed the computed `padding-bottom` is a flat `24px` regardless of context.
- **Impact:** On an iPhone with a home indicator, the last TOC item (or the Notes scratchpad, which lives at the bottom of the same drawer) could sit flush against — or be partially obscured by — the home-indicator gesture area, since the drawer treats `bottom:0` as the true edge instead of the safe area.
- **Fix direction:** Add `padding-bottom: max(var(--s6), env(safe-area-inset-bottom))` to `.toc-sidebar.mobile-open`, consistent with the pattern already used in `wiki-switcher.css:207` and `bookmarks-modal.css:154`.

### Content view — interactive (focus mode, find, LaTeX, tabs, footnotes)

#### [MAJOR] Several interactive-content text inputs render under 16px font-size — will trigger iOS auto-zoom on focus
- **Page/component:** Content view — in-article find bar, notes scratchpad (cross-cutting with Index/Preferences, see full list below)
- **File:** `css/view-content/interactive.css:222-224` (`.article-find-input`, `font-size: var(--text-sm)` = 13.6px, live-confirmed computed as `13.6px`), `css/view-content/layout.css:205-217` (`.notes-scratchpad-input`, `font-size: var(--text-xs)` = 11.5px, live-confirmed computed as `11.52px`)
- **Viewport(s):** all mobile viewports on iOS Safari specifically (WebKit-only behavior; Android Chrome doesn't auto-zoom on input focus; the underlying trigger — sub-16px computed font-size — was live-confirmed via `getComputedStyle`, but the actual auto-zoom *behavior* itself can only be observed on real iOS Safari, not Playwright/Brave)
- **Repro:** `css/tokens.css:52-54` defines `--text-xs: 0.72rem` (11.5px), `--text-sm: 0.85rem` (13.6px), `--text-base: 1rem` (16px, the iOS auto-zoom safe floor). Both listed selectors compute below 16px, live-confirmed. By contrast `.gsearch-input` (global search) correctly uses `var(--text-base)` (16px) and `.auth-field input` uses `font: inherit` resolving to the 16px body base — both safe.
- **Impact:** On iOS Safari, focusing the in-article find bar or the notes scratchpad textarea will cause the browser to auto-zoom the page in, requiring the user to manually pinch back out after dismissing the keyboard.
- **Fix direction:** Bump each to `font-size: var(--text-base)` (or an explicit `16px`) at mobile widths. See the full cross-list (including Index filter bar and Preferences data-clear select) under the "Text zoom / OS font-scaling (cross-cutting)" section below — logged once there to avoid duplication.

#### [MINOR] Glossary hover-popover (`addGlossaryTerms`) has no touch trigger — but is functionally redundant with a click-capable sibling behavior on the same element
- **Page/component:** Content view — glossary/related (glossary popovers) — see also "Content view — glossary/related" section
- **File:** `js/content/glossary-caveats.js:200-250` (`addGlossaryTerms`, mouseenter/focus/mouseleave/blur only), `js/content/glossary-caveats.js:152-198` (`addInlineGlossaryExpand`, has click + keydown), `js/render/content-view.js:399-401` (both run on the same `contentEl` back-to-back)
- **Viewport(s):** all touch viewports
- **Repro (code-read):** `addGlossaryTerms` wires only `mouseenter`/`focus`/`mouseleave`/`blur` on matched `<abbr>` elements — no `click`/`touchstart` handler, so its floating `#glossary-popover` never appears on a touch tap. However, `content-view.js` calls `addGlossaryTerms(body)` immediately followed by `addInlineGlossaryExpand(body)` on the same element set — the latter adds a `click`/`keydown` toggle that inserts an inline `.glossary-inline-def` span directly after the term. Since both run on every render, every glossary `<abbr>` ends up with a working tap-to-reveal path, even though the hover-popover half is silently dead weight on touch.
- **Impact:** No functional gap for the user — tapping a glossary term does reveal its definition via the inline-expand path. Purely a code-cleanliness/perf observation: `addGlossaryTerms`'s `IntersectionObserver` + hover wiring does real work (loads `data/glossary.json`, sets up an observer per matched term) for an affordance that's unreachable on every touch device.
- **Fix direction:** No user-facing fix needed. If revisited, consider gating `addGlossaryTerms`'s hover-popover wiring behind `matchMedia("(hover: hover)")` so it's skipped on coarse-pointer devices, since `addInlineGlossaryExpand` already covers touch.

### Topbar/nav

#### [MAJOR] Content-topbar overflow fix breakpoint (`max-width: 390px`) excludes 393px — one of this audit's four required viewports
- **Page/component:** Topbar/nav — Responsive breakpoints (cross-cutting)
- **File:** `css/responsive.css:323-341` (`@media (max-width: 390px)`), `css/responsive.css:344-349` (`@media (max-width: 375px)`)
- **Viewport(s):** 393px specifically (iPhone 14/15 Pro — one of the four viewports this audit's own matrix mandates testing)
- **Repro:** The `@media (max-width: 390px)` block's own comment states verbatim: *"content-topbar row (back + breadcrumb + read-time + 4 icon btns) overflows 390px viewports (iPhone 13) once breadcrumb hits min-width:0 - shed the read-time badge and back-btn label."* The fix (`#content-read-time { display:none }`, back-btn label hidden) only applies at `max-width: 390px`. Since 393 > 390, the iPhone 14/15 Pro viewport does **not** receive this fix.
- **Live verification (confirmed):** Tested with a real mobile Playwright context at both 390px and 393px on the same article. At 390px: `topbar.scrollWidth - topbar.clientWidth = 0` (no overflow), `#content-read-time` correctly hidden. At 393px: `topbar.scrollWidth - topbar.clientWidth = 15px` (confirmed overflow) and `#content-read-time` incorrectly still visible, crowding the row. Screenshot at 393px on the "Stack" DSA article visually confirms it: the rightmost topbar icon button is clipped/cut off at the viewport's right edge, with `< Back`, `17 min read`, and 3 full icon buttons visible before the clip.
- **Impact:** On a real iPhone 14/15 Pro (393px, arguably the single most common modern iPhone width), the content-topbar overflows by a precise, reproducible, visually-confirmed 15px — the exact bug the 390px rule was written to fix one device-width tier below. A textbook breakpoint-gap failure, and this audit's single most concrete, numerically-confirmed defect.
- **Compounding factor:** The same bug is worse under OS text-zoom — see "Text zoom / OS font-scaling (cross-cutting)" below; same fix resolves both.
- **Fix direction:** Widen the breakpoint from `max-width: 390px` to at least `max-width: 393px` (or better, `max-width: 400px`, matching the existing 400px tier used for `.prefs-tabs`, giving one consistent "narrow phone" breakpoint).

### Mobile panels / gestures

#### [MAJOR] Mobile TOC FAB button doesn't toggle closed — tapping it again while drawer is open re-opens instead of closing
- **Page/component:** Mobile panels / gestures — TOC drawer
- **File:** `js/app/mobile-panels.js:18` (`tocMobileBtn.addEventListener("click", () => openMobileToc());`)
- **Viewport(s):** 852x393 (landscape), live-verified; root cause is viewport-independent (pure JS logic bug, reproduces at any width where the FAB is shown, i.e. <=900px per `responsive.css:17-19`)
- **Repro (live-verified):** Opened an article, tapped `#toc-mobile-btn` — drawer opens correctly (`toc-sidebar mobile-open`, `body.toc-open`, overlay `.open`, confirmed via computed classes). Tapped the **same button again** expecting it to close (the button's visible state doesn't change between open/closed, so it reads as a toggle) — but `openMobileToc()` is unconditionally called again on every click, and has no closed-state branch; it only ever adds the open classes, never removes them. Confirmed: after the second tap, `tocSidebarClass`, `bodyClass`, and `overlayClass` are byte-identical to the after-first-tap state — nothing changed, the drawer silently stays open.
- **Impact:** A user who taps the TOC FAB again — the single most natural way to dismiss a drawer opened by tapping a floating button — gets no response. They're forced to discover one of the other close paths: tap the backdrop, tap a TOC item (which also navigates), swipe down, or press Escape (desktop-only). A real discoverability dead-end on a control that visually presents as a toggle.
- **Fix direction:** Change the listener to `tocMobileBtn.addEventListener("click", () => isMobileTocOpen() ? closeMobileToc() : openMobileToc());`, mirroring the toggle pattern already used correctly elsewhere in the same file (`document.getElementById("toc-collapse")`'s `classList.toggle("collapsed")` at line 14-16).

#### [MINOR] Bottom-sheet drag handles (preferences, auth, wiki switcher) are decorative only — no swipe-to-dismiss

### 404 page

#### [MAJOR] 404 page back button is completely unreachable in landscape phone orientation
- **Page/component:** 404 page
- **File:** `404.html:52-66` (`html, body { overflow: hidden; ... min-height: 100vh }`)
- **Viewport(s):** 667x375 (standard landscape phone size, live-tested)
- **Repro (live-verified):** Navigated to `404.html?title=some-nonexistent-article-name-that-is-long` in a real mobile Playwright context at 667x375 (an entirely ordinary landscape-phone dimension) with the fuzzy-suggestion block populated (3 did-you-mean results, a realistic real-world state — this repo has many stub articles). Measured: `.terminal` card height = **529.5px** against a **375px viewport**, 154.5px taller than available space. `document.body`'s `overflow: hidden` blocks all scrolling. - **Impact:** In landscape phone orientation, browsing to any not-yet-written article with 2+ fuzzy-match suggestions leaves the user with **zero way to navigate away from the 404 page** in-page — only browser chrome (URL bar, back gesture, closing the tab) escapes it. This is not a rare edge case; it's the default landscape-phone experience once suggestions populate.
- **Fix direction:** Remove `overflow: hidden` from `body` (or scope it to just the decorative `::before`/`::after` glow layers) and let the page scroll naturally when content exceeds the viewport; alternatively cap `.terminal-body` height with internal `overflow-y: auto`.

### Momentum scroll / rubber-banding vs modals (cross-cutting)

#### [CRITICAL] No body scroll-lock for search/preferences/auth/wiki-switcher/bookmarks modals or the zoom-lightbox overlay — TOC drawer is correctly locked
- **Page/component:** Global search modal, Preferences modal, Auth modal, Wiki switcher, Bookmarks modal, Content view zoom-lightbox overlay (TOC drawer is the one exception — see correction)
- **File:** `js/search.js` (`openGlobalSearch`), `js/auth.js` (`AuthModal.open`), `js/storage/settings-theme.js` (`Settings.open`), `js/app/wiki-switcher.js` (`openWikiSwitcher`), `js/app/bookmarks-modal.js`, `js/content/zoom-lightbox.js` (`openZoomOverlay`) — none of these toggle a body class or set `body.style.overflow`. Contrast with `css/responsive.css:237-241`: `@media (max-width: 900px) { body.toc-open { overflow: hidden; } }`, which `js/app/mobile-panels.js:28-34`'s `openMobileToc()` correctly triggers via `document.body.classList.add("toc-open")`.
- **Viewport(s):** 393px mobile-emulated (isMobile:true, hasTouch:true), live-verified via a real mobile Playwright context
- **Correction from initial static pass:** An initial grep-only pass searched JS files for `overflow`/`scroll-lock` patterns and wrongly concluded scroll-lock was absent everywhere, including the TOC drawer. The TOC drawer's lock is actually implemented as a CSS rule keyed off a body class, which a JS-only grep missed. **Live-verified:** opening the TOC drawer sets `document.body.className` to `"toc-open"` and `getComputedStyle(document.body).overflow` becomes `"hidden"` — confirmed working, and `window.scrollY` stayed pinned at `0` under a wheel-scroll attempt while open. The same live test on the global search modal and preferences modal showed `bodyOverflow: "visible"` and `bodyClass: ""` after opening both. The zoom-lightbox overlay (tested by opening a mermaid diagram's full-screen zoom) showed the identical gap: `bodyOverflow: "visible"` while a `position:fixed; inset:0` overlay was fully open — the single clearest case for needing a lock.
- **Impact:** On iOS Safari, the background page remains scrollable underneath an open search modal, preferences modal, auth modal, wiki switcher, bookmarks modal, or the image/diagram zoom overlay (TOC drawer is fine). A touch-drag starting on visible background content can rubber-band/scroll the page behind the modal, causing visible content shift and a "stuck" scroll position after close.
- **Fix direction:** Reuse the exact pattern already proven correct for the TOC drawer — add a body class (e.g. `body.modal-open { overflow: hidden }`) toggled by each of the other panel-open/close functions, or extract a small shared helper since the TOC drawer's approach is already the right shape, just not reused elsewhere.
